# 각 LP에 있어야 할 차별화 콘텐츠 분석 리포트

- 작성일: 2026-03-13
- 기준: Google 공식 문서 최신 기준 + 현재 저장소 LP 구조 분석
- 대상 페이지: `mortgage.html`, `apartment.html`, `refinance.html`, `business.html`
- 목적: Search AI Max, PMax, 검색광고, DB 품질 최적화 관점에서 `각 LP가 무엇으로 달라져야 하는지` 정리

## 1. 핵심 결론

### 한 줄 요약
지금 LP들은 `페이지 분리`는 되어 있지만, Google Ads 성과를 끌어올리는 데 중요한 `페이지별 고유 의사결정 콘텐츠`는 아직 부족하다.

### 현재 상태 판단
- `mortgage`, `apartment`, `refinance`, `business`는 제목, 메타, 기본 FAQ, CTA는 이미 분리되어 있다.
- 하지만 네 페이지 모두에서 공통적으로 부족한 것이 있다.
  - 표 기반 비교 콘텐츠
  - 계산기/시뮬레이터
  - 체크리스트형 자격 진단
  - 사례형 콘텐츠
  - 검색 의도에 맞는 고유 FAQ 클러스터
- 실제로 현재 4개 LP에는 `<table>`, range 기반 시뮬레이터, `체크리스트`, `비교표`, `사례` 텍스트가 확인되지 않는다.

### 검토 결론
- `같은 섹션 순서 + 같은 레이아웃 + 일부 카피만 변경`된 구조는 충분하지 않다.
- 이 정도 분화는 `페이지가 여러 장 존재한다`는 사실 자체에는 의미가 있지만, `각 페이지가 왜 따로 있어야 하는가`를 설명할 만큼 강하지는 않다.
- 특히 검색광고 기준으로는 사용자가 페이지에 들어왔을 때 `나한테 필요한 상세 정보가 있다`고 느껴야 하는데, 현재는 LP별로 그 밀도가 충분하지 않다.
- 따라서 당신의 문제 제기가 맞다. 지금 상태는 “틀린 방향”은 아니지만, `전문가용 상세 안내 페이지`로 보기에는 부족하다.

### 왜 이것이 중요한가
- Google은 AI Max에서 `text customization`, `URL inclusions`, `URL exclusions`를 제공한다.
- Google은 PMax에서 `Final URL expansion`으로 더 관련성 높은 URL을 찾고, 랜딩 내용을 바탕으로 동적 텍스트를 만들 수 있다고 설명한다.
- Search Quality Score에는 `Landing page experience`가 포함된다.
- Google은 고품질 리드 운영에서 `more qualified`와 `qualified lead` 최적화를 강조한다.

즉, LP는 단순히 제목만 다른 페이지가 아니라 `검색 의도별로 서로 다른 의사결정 도구`를 가져야 한다.

## 2. Google 공식 문서 기준 해석

### 2.1 페이지를 나눴다면, 내용도 실제로 달라야 한다
- Google이 AI Max와 PMax에서 URL과 랜딩 콘텐츠를 더 적극적으로 활용한다는 점을 보면, 각 LP는 `동일 CTA를 반복하는 복제 페이지`보다 `의도가 다른 사용자를 위해 다른 정보를 주는 페이지`여야 한다.
- 해석: LP별 고유 콘텐츠가 없으면 URL만 여러 개여도 실질 차별성이 약하다.

### 2.2 차별화 콘텐츠는 “더 많은 정보”가 아니라 “더 좋은 의사결정 도움”이어야 한다
- Google은 landing page experience와 destination quality에서 `useful`, `relevant`, `easy to navigate`를 본다.
- 해석: 사용자가 다음 행동을 결정하게 도와주는 `비교표`, `체크리스트`, `시뮬레이터`, `사례`, `FAQ`가 단순 소개 문단보다 가치가 높다.

### 2.3 리드 품질을 높이려면 LP 자체가 가벼운 qualifier 역할을 해야 한다
- Google은 고품질 리드와 lead form assets에서 `More qualified`, `qualifying question`을 강조한다.
- 해석: 웹사이트 LP도 같은 원리로 `누가 적합한지`, `어떤 경우에 가능한지`, `어떤 정보를 미리 입력해야 하는지`를 페이지 안에서 먼저 정리해야 한다.

### 2.4 아래 추천 모듈은 Google 문서의 직접 지시가 아니라 실무적 추론이다
- Google이 “아파트 LP에는 KB시세 가이드가 있어야 한다”처럼 직접 말하지는 않는다.
- 아래 제안은 Google이 강조하는 `관련성`, `유용성`, `고품질 리드`, `URL별 맥락 차별화` 원칙을 현재 대출/담보 업종에 맞게 번역한 실무안이다.

## 3. 공통 원칙: 모든 LP가 가져야 할 구조

각 LP는 아래 공통 뼈대를 공유해도 된다.
- 고유 H1
- 첫 화면 CTA
- 핵심 혜택 3개
- 기본 신뢰 요소
- 진행 절차
- 상담 폼
- 공통 법적 고지

하지만 아래 4개 블록은 LP마다 달라야 한다.
- `의사결정 도구` 1개
  - 계산기, 시뮬레이터, 자격 진단기 중 1개
- `비교 블록` 1개
  - 조건표, 경우의 수 비교표, 유형 비교표
- `고유 FAQ 클러스터` 1개
  - 4~6문항, 해당 LP 검색의도 전용
- `고유 증거 블록` 1개
  - 사례, 서류 예시, 진행 예시, 승인 패턴

실무 규칙으로는 `공통 70%, 고유 30%` 정도가 가장 안정적이다.

## 4. 페이지별 차별화 콘텐츠 권장안

## 4.1 부동산담보대출 LP (`mortgage.html`)

### 이 페이지의 역할
- 가장 넓은 상위 intent를 받는 허브 페이지
- 주택/빌라/다가구/오피스텔/토지/상가 등 다양한 담보 문의를 1차로 분류하는 페이지

### 현재 있는 것
- 광범위한 담보 유형 소개
- 기본 조건 카드
- 일반 FAQ
- 상담 CTA

### 현재 부족한 것
- 주거용 부동산 전체를 체계적으로 비교하는 표
- `1금융 vs 2금융` 차이 안내
- `담보 유형별 한도 산정 기준` 설명
- 물건 유형에 따라 어느 하위 LP로 가야 하는지 분기 가이드

### 반드시 추가할 차별화 콘텐츠
1. `주거용 부동산 유형 비교표`
- 항목: 아파트 / 빌라 / 다가구 / 단독주택 / 오피스텔
- 비교축: 시세 기준, 한도 산정 방식, 자주 필요한 서류, 후순위 가능 여부, 상담 시 확인 포인트
- 목적: 이 페이지를 “종합 안내 허브”로 만들기

2. `시세 기준별 한도 산정 가이드`
- 항목: KB시세 / 감정가 / 실거래가 참고 / 금융사 내부 평가
- 주의: 실제 숫자 기준은 자주 변할 수 있으므로 `정확 수치 확정표`보다 `산정 로직 가이드` 형태가 안전하다

3. `1금융 vs 2금융 차이 안내`
- 항목: 심사 관점, DSR 영향, 속도, 추가 한도 가능성, 대환 적합 상황
- 목적: 검색 사용자가 왜 상담이 필요한지 이해하게 만들기

4. `담보 유형 선택형 분기 블록`
- “아파트면 apartment LP”, “대환/후순위면 refinance LP”, “사업자/상가/공장이면 business LP”로 이동시키는 중간 허브 역할

### Google Ads 관점에서 왜 필요한가
- 이 페이지는 broad intent를 받아서 `다음 페이지로 분기시키는 허브`가 되어야 한다.
- broad query를 받는 페이지인데도 구체 비교가 없으면 `광고는 넓게 받고 랜딩은 얕은` 상태가 된다.

### 우선순위
- 우선순위 1: 주거용 유형 비교표
- 우선순위 2: 1금융 vs 2금융 차이 안내
- 우선순위 3: 시세 기준별 한도 산정 가이드

## 4.2 아파트담보대출 LP (`apartment.html`)

### 이 페이지의 역할
- 가장 전환 친화적인 구체 intent 페이지
- “아파트”, “KB시세”, “후순위”, “대환” 키워드를 흡수하는 상세 LP

### 현재 있는 것
- KB시세 언급
- 선순위/후순위 기본 구분
- 기본 FAQ
- CTA

### 현재 부족한 것
- KB시세 조회 안내 프로세스
- 전용면적/기존 대출 잔액 기반 예상 한도 계산 보조
- 선순위/후순위/대환 3자 비교표
- 지역/규제 여부 차이를 다루는 설명 블록

### 반드시 추가할 차별화 콘텐츠
1. `KB시세 조회 안내 블록`
- 항목: 주소 확인 → KB시세 확인 → 기존 대출 잔액 반영 → 예상 가능 범위 안내
- 목적: 사용자가 “무엇을 준비하면 상담이 빨라지는지” 알게 하기

2. `아파트 예상 한도 시뮬레이터`
- 입력 예시: KB시세, 기존 1순위 잔액, 희망 자금, 대환 여부
- 출력 예시: 예상 추가 한도 범위, 상담 시 확인 필요 항목
- 주의: 확정 금액처럼 보이지 않게 `예상 범위`로 표현

3. `선순위 / 후순위 / 대환 비교표`
- 항목: 어떤 상황에서 적합한지, 기존 대출 유지 여부, 금리 절감 목적 여부, 추가 한도 목적 여부, 필요 확인 항목

4. `지역/규제 차이 안내`
- 항목: 규제지역 여부, 금융사별 심사 차이, 아파트 유형별 예외
- 주의: 수시 변하는 숫자를 박제하지 말고 `정책/규제/금융사 기준에 따라 변동 가능` 문구를 함께 둔다

### 추가하면 좋은 콘텐츠
- `전용면적별 상담 포인트`
  - 소형 / 중형 / 대형 평형대에 따른 시세 확인 포인트
- `주소만 있으면 확인 가능한 항목` 요약
  - KB시세
  - 기존 대출 여부
  - 후순위 가능성

### Google Ads 관점에서 왜 필요한가
- apartment LP는 Search 광고에서 가장 query match가 명확한 페이지다.
- 따라서 이 페이지는 “설명”보다 “계산과 비교”에 강해야 한다.
- AI Max/PMax가 이 페이지로 보내는 이유를 강화하려면 `아파트 전용 계산/비교 모듈`이 필요하다.

### 우선순위
- 우선순위 1: 선순위/후순위/대환 비교표
- 우선순위 2: KB시세 조회 안내
- 우선순위 3: 예상 한도 시뮬레이터
- 우선순위 4: 지역/규제 차이 안내

## 4.3 대환/후순위 LP (`refinance.html`)

### 이 페이지의 역할
- “고금리 → 저금리”, “이자 절감”, “기존 대출 유지”, “추가 자금” 같은 문제 해결형 검색의도를 받는 페이지

### 현재 있는 것
- 선순위 대환 / 후순위 추가 / 후순위 대환 구분
- 대환 설명 FAQ
- 기본 조건 안내

### 현재 부족한 것
- 대환 효과를 바로 체감시키는 계산기
- 대환 전/후 비교 사례
- 대환 가능 조건 체크리스트
- “나는 선순위 대환이 맞는지, 후순위 추가가 맞는지” 판단 도구

### 반드시 추가할 차별화 콘텐츠
1. `이자 절감 시뮬레이터`
- 입력 예시: 기존 금리, 기존 잔액, 잔여 기간, 예상 대환 금리
- 출력 예시: 월 이자 차이, 연간 절감액, 상담 시 확인 항목
- 목적: 대환의 가치를 숫자로 체감시키기

2. `대환 전 / 대환 후 비교 사례 카드`
- 항목: 기존 금리, 대환 후 금리, 추가 한도 확보 여부, 상담 포인트
- 주의: 실제 고객 데이터라면 익명화와 과장 금지

3. `대환 가능 조건 체크리스트`
- 항목: 기존 대출 종류, 현재 금리 수준, 연체 여부, 담보 잔여 가치, 추가 자금 필요 여부
- 형식: 5문항 이하의 짧은 yes/no형 진단

4. `선순위 대환 vs 후순위 추가 vs 후순위 대환 선택 가이드`
- 이 페이지에서 가장 중요한 비교 블록
- 사용자가 어떤 상담 유형을 고르면 되는지 바로 알 수 있어야 한다

### 추가하면 좋은 콘텐츠
- `손익분기 시점` 안내
  - 중도상환수수료나 부대비용이 있는 경우 언제부터 이익인지
- `대환 시 자주 막히는 이유` FAQ
  - 연체
  - 담보 잔여 가치 부족
  - 서류 미비

### Google Ads 관점에서 왜 필요한가
- refinance LP의 핵심 intent는 정보 탐색이 아니라 “손해를 줄일 수 있는지 판단”이다.
- 따라서 이 페이지는 네 LP 중 `계산기 효과`가 가장 큰 페이지다.
- 이자 절감 계산기와 전/후 비교 사례는 `more qualified` 리드로 이어질 가능성이 높다.

### 우선순위
- 우선순위 1: 이자 절감 시뮬레이터
- 우선순위 2: 선순위/후순위/대환 선택 가이드
- 우선순위 3: 대환 전/후 사례 카드
- 우선순위 4: 대환 가능 조건 체크리스트

## 4.4 사업자 LP (`business.html`)

### 이 페이지의 역할
- 사업자, 법인, 비주거용 부동산, 고액 담보 문의를 받는 페이지
- 토지/상가/공장/오피스텔/법인 명의 등 복잡한 intent를 구조화하는 페이지

### 현재 있는 것
- 토지/상가/공장/오피스텔 유형 노출
- 개인사업자/법인사업자 기본 구분
- FAQ
- CTA

### 현재 부족한 것
- 물건 유형별 상세 조건 비교표
- 개인사업자 vs 법인 필요 서류 리스트
- 사업자등록 기간별 가능성 가이드
- 산업용 물건 특화 FAQ

### 반드시 추가할 차별화 콘텐츠
1. `담보 물건 유형별 상세 조건 비교표`
- 항목: 토지 / 상가 / 공장 / 오피스텔 / 창고
- 비교축: 평가 기준, 자주 보는 서류, 상담 시 핵심 확인 포인트, 금융사 선호도 차이

2. `개인사업자 vs 법인 필요 서류 리스트`
- 최소한 2열 비교 구조로 분리
- 개인사업자: 사업자등록증, 소득/매출 확인자료 등
- 법인: 법인등기부등본, 재무제표, 주주구성/대표자 관련 추가 확인 등

3. `사업자등록 기간별 가능 여부 가이드`
- 예시 구조: 신설 / 6개월+ / 1년+ / 업력 충분
- 주의: 실제 승인 기준은 금융사별로 다르므로 `가능성 가이드`로 표현하고 확정표는 피한다

4. `용도별 FAQ 클러스터`
- 토지
- 상가
- 공장
- 임대수익형 오피스텔
- 법인 명의

### 추가하면 좋은 콘텐츠
- `사업자 자금 용도 예시`
  - 운영자금
  - 대환
  - 시설투자
  - 세금/정산 자금
- `비주거용 담보 상담 전 준비사항`
  - 주소
  - 물건 종류
  - 사업자 형태
  - 기존 대출 여부

### Google Ads 관점에서 왜 필요한가
- business LP는 검색어 폭은 넓지만 조건이 복잡한 페이지다.
- 따라서 이 페이지는 “상담 유도형 카피”만으로는 부족하고, `복잡한 조건을 정리해주는 구조화 콘텐츠`가 강해야 한다.
- 사업자/법인 문의는 리드 품질 편차가 크므로, 체크리스트와 서류 안내가 lead quality를 높인다.

- 우선순위 3: 사업자등록 기간별 가능성 가이드

## 6. 구현 시 주의사항

### 6.1 LP마다 똑같은 FAQ를 복붙하지 말 것
- Google Ads와 SEO 모두 관점에서 `URL만 다른 유사 페이지`는 강점이 약하다.
- 공통 FAQ는 일부만 공유하고, 각 LP마다 4~6개는 전용 질문을 두는 것이 좋다.

### 6.2 숫자형 규제 정보는 박제하지 말 것
- LTV, 지역 규제, 금융사 승인 조건은 변동 가능성이 있다.
- 따라서 고정 숫자 표보다 `산정 방식`, `영향 변수`, `상담 시 확인 포인트` 중심으로 쓰는 편이 안전하다.
- 최신 수치를 노출하려면 운영팀이 갱신 책임을 가져야 한다.

### 6.3 계산기/체크리스트는 반드시 데이터로 남길 것
- 계산기 입력값이나 체크리스트 응답은 단순 UX 요소로 끝내지 말고 DB에 남겨야 한다.
- 예시:
  - 아파트: KB시세, 기존 잔액, 대환 여부
  - 대환: 현재 금리, 잔액, 희망 자금
  - 사업자: 물건 유형, 사업자 형태, 업력
- 이렇게 해야 `raw lead`보다 더 좋은 `qualified lead` 신호를 만들 수 있다.

### 6.4 한 페이지에 차별화 요소를 과도하게 넣지 말 것
- LP당 가장 강한 인터랙티브 모듈 1개면 충분하다.
- 나머지는 표, FAQ, 체크리스트처럼 가볍고 빠르게 읽히는 블록으로 보완하는 편이 전환율에 유리하다.

## 8. 최종 제안

이 프로젝트의 LP 전략은 이미 방향은 맞다.  
문제는 `페이지 수`가 아니라 `페이지별 의사결정 콘텐츠 밀도`다.

다음 단계에서 필요한 것은 새 페이지를 더 만드는 것보다, 이미 만든 네 LP를 아래 방식으로 진짜 다르게 만드는 것이다.
- `mortgage`: 허브형 비교
- `apartment`: 계산형 비교
- `refinance`: 절감형 계산
- `business`: 구조화 정보형 비교

네 페이지가 이 역할로 분화되면, Search AI Max와 PMax가 URL을 활용할 이유가 더 분명해지고, 사용자는 더 빠르게 자기 상황에 맞는 상담으로 들어오게 된다.

## 9. 실제로 더 좋은 LP를 만들기 위해 당신이 가져오면 좋은 정보

아래 정보가 있으면, 지금 문서를 `실제 페이지용 표/계산기/FAQ/체크리스트 초안`으로 바로 바꿀 수 있다.

### 9.1 `mortgage.html`용
- 실제 취급 가능한 담보 유형 목록
  - 주택, 빌라, 다가구, 단독, 오피스텔, 토지, 상가, 공장 중 어디까지 진짜 가능한지
- 유형별 한도 산정 기준
  - KB시세, 감정가, 실거래 참고, 내부 평가 중 무엇을 주로 쓰는지
- 1금융/2금융/저축은행/캐피탈을 비교할 때 실제 상담에서 설명하는 차이
- “이런 경우는 어렵다”는 제외 케이스

### 9.2 `apartment.html`용
- 실제 KB시세 확인 절차
  - 고객에게 주소만 받는지, 평형/동호수까지 필요한지
- 선순위/후순위/대환 중 어떤 질문으로 분기하는지
- 규제지역/비규제지역 또는 지역 차이를 내부에서 어떻게 설명하는지
- 상담 시 가장 많이 나오는 질문 10개

### 9.3 `refinance.html`용
- 실제 대환 전/후 사례 3~5개
  - 익명화된 숫자면 충분
- 대환 상담에서 꼭 확인하는 변수
  - 현재 금리, 잔액, 잔여기간, 중도상환수수료, 추가자금 필요 여부
- 어떤 경우에 선순위 대환이 맞고, 어떤 경우에 후순위 추가가 맞는지 내부 판단 기준
- “대환이 안 맞는 경우” 예시

### 9.4 `business.html`용
- 실제 취급 물건 우선순위
  - 토지, 상가, 공장, 오피스텔, 창고, 임야 중 무엇이 강한지
- 개인사업자 vs 법인사업자 필요 서류 실제 목록
- 사업자등록 기간에 따른 상담 가능 기준
  - 신설 가능 여부, 6개월+, 1년+ 같은 내부 기준
- 산업/업종별로 어려운 경우

### 9.5 가져오면 안 되는 정보
- 오래된 금리표
- 출처 불명 LTV 숫자
- 특정 금융사 최신 조건을 확정표처럼 적은 자료

이런 정보는 변동성이 크기 때문에 페이지에 박제하면 오히려 신뢰를 해칠 수 있다.
