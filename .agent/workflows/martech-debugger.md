# 🕵️‍♂️ 마케팅 데이터 체계적 디버깅 (MarTech Systematic Debugging)

마케팅 데이터(구글 애즈 전환, GA4 이벤트 누락, 페이스북 픽셀 유실 등)가 제대로 찍히지 않을 때, "감으로 스크립트 위치를 바꿔보거나" "아무 설정이나 껐다 켜보는" 행위는 절대 금물입니다.
이 워크플로우는 백엔드 개발자들의 **'체계적 디버깅(Systematic Debugging)'** 방법론을 마케팅 트래킹 환경에 맞춰 완벽하게 진화시킨 추적 프로세스입니다.

## 🚨 철칙 (The Iron Law)
> **"원인이 어느 구간에 있는지 좁히기 전까지, 라이브 서버의 코드나 태그를 함부로 변경하지 않는다."**

## 🚦 디버깅 4단계 (The Four Phases)

### Phase 1: 파이프라인 구간 분리 및 데이터 추적 (Root Cause Investigation)

마케팅 데이터는 여러 파이프라인을 거칩니다. 어디서 끊겼는지 구간부터 찾아야 합니다.

1. **상태 재현하기**
   - 직접 시크릿 모드를 켜고 구글 애즈 테스트 파라미터(`?gclid=test`)를 붙여서 접속해 봅니다.
   - 버튼 클릭부터 '완료 페이지' 도달까지 페이지 이동 시 파라미터가 유실되지 않고 주소창에 계속 살아있는지 확인합니다.
2. **구간별 생존 확인 (증거 수집)**
   - **구간 A (프론트엔드 버튼):** F12(개발자 도구) > Network 탭을 열고 버튼을 누를 때 `google`이나 `analytics`로 신호(Payload)가 제대로 쏘아지는지 확인.
   - **구간 B (GTM/태그 매니저):** GTM '미리보기(Preview)' 모드에서 트리거가 정상적으로 'Fired(실행됨)' 되는지 검증.
   - **구간 C (수집 데이터):** GA4의 **[관리] > [DebugView]** 메뉴를 열어두고 내가 방금 누른 클릭이 실시간으로 도달했는지 확인.

### Phase 2: 패턴(원인) 분석 (Pattern Analysis)

문제가 발생하는 구간을 찾았다면, 이유를 분석합니다. 마케팅 테크 오류의 90%는 아래 3가지 패턴 중 하나입니다.

1. **리다이렉트 충돌 패턴 (가장 흔함)**
   - 버튼을 누르자마자 다음 페이지로 빠르게 튕겨버리면, 브라우저가 구글에 데이터를 전송할 0.1초의 시간조차 없어서 데이터가 유실됩니다.
2. **비동기 로딩 스크립트 패턴**
   - 팝업이나 모달창 안에 있는 버튼이라, 페이지 접속 시점에 구글 스크립트가 버튼 자체를 인식하지 못했을 가능성 파악.
3. **UTM/GCLID 유실 패턴**
   - 페이지 1에서 페이지 2로 넘어갈 때, 링크 설정이 잘못되어 꼬리표가 잘려 나가지 않았는지 점검.

### Phase 3: 가설 수립 및 최소 테스트 (Hypothesis & Testing)

1. **단일 가설 세우기**
   - ❌ "스크립트 위치를 헤더에서 바디로 내리고, 트리거 조건을 바꿔보자" (동시 변경 금지)
   - ⭕ "페이지가 너무 빨리 넘어가서 데이터 전송이 취소됐을 것이다. 버튼 클릭 시 0.5초(500ms) 딜레이를 주면 해결될 것이다."
2. **안전한 격리 테스트**
   - 스크립트 관리자(GTM)에서 테스트 버전만 저장한 상태로(미게시) 혼자서 쏴봅니다.

### Phase 4: 해결 및 검증 (Implementation)

1. **해결책 적용**
   - 가설이 맞았다면, 예를 들어 프론트엔드 코드에 `setTimeout()`을 적용하거나, GTM의 '태그 실행 대기' 브라우저 옵션을 활성화합니다.
2. **최종 교차 검증 (방어적 디버깅)**
   - 구글 환경(GA4)에서만 체크하지 않고, 실제 서버의 DB에 쌓인 고객 시간대와 GA4에 찍힌 시간대를 교차 검증하여 누락율을 2% 미만으로 좁힙니다.
3. **재발 방지 문서화**
   - 왜 유실되었는지, 어떻게 방어 로직을 짰는지 코멘트를 남겨 같은 방식으로 마케팅 비용이 누수되지 않게 합니다.
